## Defaults applied (no separate Redux/CRA ‚Äî flag if you object)

- **Build tool: Vite** (React 19; CRA is deprecated). Spec's `main.jsx` + `index.html`
  layout is Vite convention. **Deviation:** Vite puts `index.html` at the `frontend/`
  root, not under `public/` as the spec layout shows; `public/` keeps static assets
  (`favicon.ico`, `logo.png`). Documented in `ROUTING_DOCUMENTATION.md`/README.
- **Token handling (C-08/C-09):** JWT held **in memory** (Context) only ‚Äî never in
  `localStorage`/`sessionStorage` (C-09). Documented limitation: production should move
  to an httpOnly cookie issued by the backend; noted in `API_INTEGRATION_GUIDE.md`.
- **Binary output files deferred:** `TEST_CASES.xlsx` ‚Üí produced as `TEST_CASES.md`
  (markdown table; xlsx export is a manual step, flagged). `SCREENSHOTS/*.png` ‚Üí
  captured manually after the app runs; `SCREENSHOTS/README.md` placeholder created.
  `build/` is generated by `npm run build` (gitignored, not committed).
- **Styling:** plain CSS modules + a small global `index.css` (no extra UI framework)
  to keep dependencies minimal and the build fast (AC-16). Flag if you want a UI kit.

## Assumed REST API contract (documented in `API_INTEGRATION_GUIDE.md`, mocked now)

Base: `${VITE_API_BASE_URL}` (default `http://localhost:8000`). All JSON.

| Area | Method & path | Purpose |
| :-- | :-- | :-- |
| Auth | `POST /api/auth/register` | `{name,email,password,role:"customer"|"vendor"}` ‚Üí `{token,user}` |
| Auth | `POST /api/auth/login` | `{email,password}` ‚Üí `{token,user}` |
| Auth | `GET /api/auth/me` | (Bearer) ‚Üí `{user}` |
| Products | `GET /api/products?query=&page=` | list `{id,name,price,vendor,rating,availability,stock,category}` |
| Products | `GET /api/products/:id` | one product |
| Products | `POST/PUT/DELETE /api/products[/:id]` | vendor CRUD (AC-13/14/15) |
| Search | `GET /api/search?q=` | results with Name, Price, Vendor, Rating, Availability (AC-10) |
| Chatbot | `POST /api/chat` | `{message,sessionId}` ‚Üí `{reply,listings?}` (AC-11) |
| Orders | `GET /api/orders` / `POST /api/orders` | list / place (multi-vendor cart ‚Üí one order number, per SPEC ¬ß3) |

> This contract is the frontend's working assumption; it is **not** a backend
> commitment. Marked `[NEEDS CLARIFICATION: backend to confirm/replace]` in the guide.

### Mock approach (D5 ‚Äî clearly not real vision)
- `extract/product`: parse a `prompt` for a price number / known category / name; from an
  `image`, derive a candidate name from the filename + guess category. Returns
  `{ product: { name, price, stock, category, description } }`.
- `search/image`: derive a keyword from the image filename; reuse cheapest-first search;
  if no keyword match, return top in-stock items as "visually similar".
